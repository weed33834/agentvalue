# ADR-001:IM 集成方案选型(飞书优先)

- **状态:** 已接受(Accepted)
- **日期:** 2026-07-03
- **决策者:** 集成架构组
- **关联:** Phase 7.2 IM 集成

## 上下文

AgentValue-AI 的评估输入目前主要靠员工在前端手填日报,外加 mock 数据跑通链路。要走向真实试点,日报的**自动采集**是绕不开的一环——靠人手填,首周活跃率就会塌。企业 IM 是日报最自然的承载体:员工本来就在群里发日报,让机器人接一下就能进系统,几乎零额外操作。

候选三家:飞书、钉钉、企业微信。选型要权衡四件事:

1. **市场占有率**:决定能覆盖多少试点团队。钉钉用户基数最大(传统行业尤其强),飞书在互联网/新经济企业占优,企业微信偏私域和客户运营场景。
2. **API 成熟度**:机器人接收消息、事件订阅、回调签名的文档与稳定性,直接决定接入摩擦。
3. **开发成本**:能否复用现有 `/api/v1/inputs` 端点,而不是新开一套入库链路。
4. **数据合规**:企业版是否提供完整审计日志,满足评估系统"可追溯可解释"的硬要求。

约束:AgentValue-AI 已有 `POST /api/v1/inputs` 端点(`backend/api/routes.py:219`),接受 `employee_id/period/type/content/attachments`,输入护栏与审计日志都已接入,且 `type` 是自由字符串(默认 `daily_report`)。集成方案应当**复用这个端点**,而不是另起炉灶——否则护栏、审计、状态机都要重做一遍。

## 决策

**首期选飞书,钉钉作为二期,企业微信暂不排期。**

理由是实在的取舍,不是"飞书最好"这种空话:

- **飞书开放平台文档结构清晰、API 风格统一**(REST + JSON,事件订阅模型规范)。接收群消息走自建应用的机器人能力 + 事件订阅(`im.message.receive_v1`),回调地址收到消息后转发到 `/inputs`,链路短、坑少。
- **企业版审计日志完整**。日报采集涉及员工沟通内容进入评估系统,必须有可追溯的采集记录。飞书企业版管理后台的审计能力能满足这条硬约束,无需额外自建。
- **能直接复用 `/inputs`**。机器人回调适配层只做格式转换(飞书消息 → `CreateInputRequest`),`type=daily_report`、`content=消息正文`,护栏与审计自动生效,不动主链路。
- **二期再上钉钉是经济账**。钉钉用户基数更大,但机器人事件订阅文档新旧两套并存(OAPI 旧版与新版混用),首期接入摩擦明显,且完整审计需旗舰版。先把飞书跑通、验证采集→评估闭环,再按试点团队实际使用的 IM 决定二期优先级,比一开始铺三家稳。

## 集成方式

不新增端点,复用 `/api/v1/inputs`。新增一个轻量适配层(部署为独立小服务或 AgentValue-AI 内的 adapter 模块;Phase 7.2 只定契约,不写代码):

```
飞书群(员工发日报)
   │  消息事件 im.message.receive_v1
   ▼
飞书自建应用机器人(事件订阅回调)
   │  POST 回调地址(带 X-Lark-Signature 签名校验)
   ▼
AgentValue-AI IM 适配层
   │  格式转换 + 服务态鉴权(系统级 JWT / 内部 API Key)
   │  构造 CreateInputRequest{type=daily_report, content=消息正文}
   ▼
POST /api/v1/inputs
   │  输入护栏 → 审计日志 → RawInput 入库
   ▼
评估流(按 period 触发)
```

几个工程上必须说清楚的点:

- **接收消息不是"自定义机器人 webhook"**。飞书自定义机器人本质是**单向推送**(往群里发消息),接收消息必须用自建应用的机器人能力 + 事件订阅。这一点容易踩坑,先记下,落地时按自建应用走。
- **回调鉴权双向**。飞书侧用 `X-Lark-Signature` 校验回调来源;AgentValue-AI 侧,适配层调 `/inputs` 时需带服务态凭证——`/inputs` 有 `require_role` 鉴权,机器人没有员工身份,必须用系统级 JWT 或内部 API Key,不能直接裸调。
- **员工身份映射**。飞书 `open_id` → AgentValue-AI `employee_id` 需要一张映射表,在适配层维护,不污染主链路。

## 后果

**正面:**

- 日报采集闭环用最小改动跑通,主链路护栏/审计/状态机全部复用。
- 飞书试点团队首周就能用上自动采集,活跃率有保障。

**负面 / 需接受:**

- 首期只支持飞书。钉钉/企业微信为主的企业,试点前要么手动导入日报,要么等二期。
- 飞书开放平台 API 变更(事件版本升级、字段调整)需要持续跟进,适配层是唯一耦合点,变更影响收敛在这里。
- 适配层是新增组件,需要单独的部署、监控与故障兜底(回调积压、签名失败重试)。这部分进 Phase 7.2 之后的实现任务,不在本 ADR 范围。

## 备选方案

| 方案 | 评估 | 不选的原因 |
|---|---|---|
| **钉钉** | API 也成熟,用户基数最大,事件订阅能跑通 | 机器人事件订阅文档新旧两套混用,首期接入摩擦大;完整审计需旗舰版;作为二期更稳妥 |
| **企业微信** | 群机器人为主、生态偏腾讯内部 | 机器人能力偏 push-only,接收消息需自建应用回调,文档相对单薄;员工日报场景市场占比第三,投入产出比低 |
| **自建 webhook 网关** | 完全灵活,IM 无关 | 开发量大,要自管鉴权/重试/多 IM 协议适配,等于自造一个 IM 中台;MVP 阶段不值 |
| **三家同时上** | 覆盖最广 | 首期就铺三套适配层,验证成本高、回归面大;先验证一家再扩,风险更可控 |
